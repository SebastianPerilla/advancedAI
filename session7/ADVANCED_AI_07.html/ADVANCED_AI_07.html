The YOLOv5 repository includes several main scripts, but two of the most important ones are train.py and detect.py. These scripts are fundamental for training and using YOLOv5 models.

a) detect.py

  • Purpose: The detect.py script is used to apply a trained YOLOv5 model for object detection on images, videos, or live camera feeds. It's useful for inference and showcasing the model's capabilities. We could directly use it for the same purpose as before.

  • Usage:

    • detect.py --source specifies the source for inference, which can be an image, video file, or camera feed.
    • detect.py --weights points to the trained YOLOv5 model checkpoint to use for inference.
    • detect.py --img-size sets the input image size for inference.
    • detect.py --conf-thres sets the confidence threshold for object detection.
    • detect.py --iou-thres sets the IoU (Intersection over Union) threshold for non-maximum suppression.
    • detect.py --device specifies the device (CPU or GPU) for inference.

  • Explanation: detect.py loads a trained YOLOv5 model and applies it to the specified input source. It detects objects in images or frames of a video stream and can be used for various applications, including autonomous vehicles, surveillance, and more.

b) train.py

  • Purpose: The train.py script is used to train a YOLOv5 model on a custom dataset. It's a crucial component for fine-tuning or creating our own object detection models.

  • Usage:

    • train.py --img-size specifies the input image size for training.
    • train.py --batch-size sets the batch size for training.
    • train.py --epochs determines the number of training epochs.
    • train.py --data specifies the path to the data.yaml configuration file containing dataset information.
    • train.py --weights allows to start training from a pre-trained YOLOv5 model checkpoint.
    • train.py --cfg specifies the YOLOv5 model configuration file to use.
    • train.py --name sets a name for the training experiment.
    • train.py --project specifies the project name for logging.

  • Explanation: train.py takes care of loading the dataset, initializing the YOLOv5 model architecture, and performing the training loop. It updates the model's weights to minimize the detection loss, making it suitable for custom object detection tasks.

These two main scripts, train.py and detect.py, form the core of YOLOv5's capabilities, allowing to both train custom models and apply them for object detection tasks. Understanding these scripts is essential for anyone studying computer vision and deep learning.

If we take a look at the images after the detection, we can see that some of them have 2 boxes, each predicting 1 class.

Because wolves and foxes are visually similar (shape, pose, environment, etc.) the model may give high confidence scores for both classes in the same region.

For a given object, you might see something like:

Box Objectness Fox score Wolf score
A 0.9 0.85 0.75

YOLOv5 then reports the highest class score per box by default, but it might still generate separate overlapping boxes for two classes if both are confident and survive non-max suppression (NMS).

YOLOv5 uses class-aware NMS by default, meaning:

  • It suppresses overlapping boxes within the same class.
  • But it does not suppress overlapping boxes of different classes. So, if the model predicts:
    • Box A (fox, 0.85)
    • Box B (wolve, 0.75)

and they overlap heavily, both will remain, because NMS only removes duplicates of the same class

We could fix this using:

!python yolov5/detect.py --weights yolov5/runs/train/exp/weights/best.pt --source fox_and_wolves/test/images/ --agnostic-nms

This makes NMS class-agnostic, meaning that if two boxes overlap a lot, only the highest-confidence one remains, regardless of class.

What could we do to improve the results?

  • Add more diverse training images different poses, lighting, distances, seasons, etc.
  • Include hard negatives (images with no foxes or wolves) so the model learns when not to predict anything.
  • Check for mislabelled or inaccurate bounding boxes (common source of noisy predictions).
  • Increase the number of epochs

  • GPU/CPU usage
  • Model size (.pt file size)

Multi-class fine-tuning YOLO for better performance

If we wanted our model to detect both COCO classes and foxes and wolves (new classes), we'd need to do a multi-class fine-tunning

For this, we'd need to have some COCO data as well, or some images with all those classes, plus some images with foxes and wolves

We would also need to update our yaml file to include the 81 classes that we will have now (COCO has 80)

In this case it would be pretty important that, if we don't want to miss the performance on the existing COCO classes, that both our training dataset and validation datasets are well balanced.

The training process would be the same

!python yolov5/train.py \
    --img 640 \
    --batch 16 \
    --epochs 3 \
    --data fox_and_wolves_and_COCO/data.yaml \
    --weights yolov5s.pt \
    --cache

Or we could also freeze some layers
